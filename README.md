Texto 2
A engenharia de software é definida como "programação integrada ao longo do tempo". Não basta apenas o código funcionar hoje; ele precisa ser sustentável. O foco está em três pilares:
Tempo e Mudança: O código deve ser capaz de evoluir e ser mantido por anos.
Escala e Crescimento: Como a organização e o sistema lidam com o aumento de usuários e dados.
Trade-offs: Como tomar decisões baseadas em custos e benefícios.

3 Exemplos de Trade-offs
Um trade-off ocorre quando você precisa abrir mão de uma qualidade para ganhar outra. Na engenharia, raramente existe uma solução perfeita, apenas a "menos pior" para aquele momento.

1. Velocidade de Entrega vs. Qualidade do Código (Dívida Técnica)
O que é: Decidir lançar um recurso rapidamente para o mercado usando um código "bagunçado" ou gastar mais tempo para escrever um código limpo e testado.

A troca: Você ganha tempo agora, mas perde sustentabilidade, pois terá mais trabalho para consertar ou atualizar esse código no futuro.

2. Consistência vs. Disponibilidade (Teorema CAP)
O que é: Em sistemas distribuídos (como grandes bancos de dados), às vezes você precisa escolher entre todos os usuários verem o dado exato ao mesmo tempo ou o sistema nunca sair do ar.

A troca: Se você prioriza a disponibilidade, o sistema responde rápido, mas um usuário pode ver uma informação levemente desatualizada por alguns segundos.

3. Segurança vs. Usabilidade
O que é: Adicionar várias camadas de proteção, como autenticação de dois fatores (2FA), senhas complexas e expiração de sessão curta.

A troca: Você aumenta a segurança, mas diminui a usabilidade. O usuário fica mais protegido, porém a experiência de uso se torna mais lenta e burocrática.
